# Adapter Contract

## 1. Purpose

This document defines how MVP adapters for Codex, Claude, Copilot, and MCP must behave. The goal is to keep vendor-specific logic isolated while preserving one shared planning model.

## 2. Adapter role

An adapter is responsible for translating normalized manifest intent into target-specific files. Adapters must not own repo detection, defaulting, or raw prompt collection.

The required flow is:

1. manifest loader normalizes config
2. planner computes the output plan
3. adapters validate support and render file contents
4. file writer handles diffing and writes

## 3. Core rules

- Adapters must consume normalized manifest input only.
- Adapters must write only files assigned to them by the planner.
- Adapters must not perform direct filesystem writes.
- Adapters must return warnings for unsupported or downgraded behavior.
- Adapters must produce deterministic output for the same input.

## 4. Interface

The MVP adapter interface should be richer than the minimal TRD sketch so planning and diagnostics stay explicit.

```ts
type AdapterId = "codex" | "copilot" | "claude" | "mcp";

type SupportIssue = {
  severity: "warning" | "error";
  code: string;
  message: string;
};

type SupportResult = {
  supported: boolean;
  issues: SupportIssue[];
};

type RenderedFile = {
  path: string;
  content: string;
  trustSensitive: boolean;
};

interface Adapter {
  id: AdapterId;
  supports(manifest: Manifest): SupportResult;
  plan(manifest: Manifest, plan: GenerationPlan): PlannedFile[];
  render(file: PlannedFile, manifest: Manifest): RenderedFile;
}
```

The planner remains the source of truth for destination paths. Adapter `plan()` may filter or annotate only the files owned by that adapter.

## 5. Required inputs

Each adapter must receive:

- normalized `Manifest`
- shared `GenerationPlan`
- the target-owned `PlannedFile`

Adapters must not reach back into raw wizard answers or repo inspection output directly.

## 6. Planning responsibilities

During planning, an adapter must:

- confirm whether the selected target is supported
- identify unsupported feature combinations
- attach target-specific warnings
- avoid expanding beyond the documented output map

Planning must not produce file content yet.

## 7. Rendering responsibilities

During rendering, an adapter must:

- render stable content for one planned file at a time
- include generated headers when format permits
- mark trust-sensitive outputs
- avoid including secrets or machine-specific values directly

Rendering must be a pure operation from inputs to output text.

## 8. Warning model

Warnings should be categorized so the CLI can present them consistently.

Recommended warning codes:

- `unsupported_target_feature`
- `downgraded_generation_mode`
- `local_file_required`
- `manual_followup_required`
- `trust_sensitive_output`

Warnings must explain:

- what was requested
- what the adapter can actually do
- what file or target is affected
- what the user should do next

## 9. Unsupported mapping behavior

Unsupported mappings must degrade gracefully.

### Required behavior

- Skip only the unsupported file group.
- Continue planning for the rest of the target when possible.
- Emit at least one warning with a concrete reason.
- Do not silently drop requested behavior.

### Examples

- If prompt packs are requested for a target that only supports base instructions, keep base output and warn that prompts were skipped.
- If a target can support instructions but not agent templates, generate instructions and warn that agent files were omitted.
- If an MCP preset requires secrets not expressible safely in shared config, route output to local scope or warn and skip.

## 10. Determinism requirements

Adapters must produce deterministic output across repeated runs:

- stable heading order
- stable bullet order
- stable frontmatter key order when used
- stable newline behavior

Adapters must not embed timestamps, random IDs, or machine-local paths in shared outputs.

## 11. Generated header requirements

When the target format allows it, adapters should include a short generated notice such as:

```text
Generated by agenv. Review before editing by hand.
```

The exact syntax may vary by file type, but the writer must be able to recognize the header later for safe overwrite behavior.

## 12. Target-specific expectations

### Codex adapter

Responsibilities:

- render `AGENTS.md`
- reflect repo conventions, review rules, and dashboard workflow guidance
- optionally reference shared prompt docs, but not depend on them

Must not:

- invent Codex-only schema outside `AGENTS.md`
- emit local config files in MVP

### Copilot adapter

Responsibilities:

- render `.github/copilot-instructions.md`
- tailor instruction phrasing for repo-level Copilot customization

Must not:

- rely on unsupported subagent or hook concepts
- emit files outside the documented Copilot path

### Claude adapter

Responsibilities:

- render base Claude workspace docs
- render skill templates when enabled
- render agent templates when enabled
- render local Claude settings only when local scope is needed

Must not:

- assume hooks are enabled by default
- write executable hooks unless explicitly supported later

### MCP adapter

Responsibilities:

- render MCP config files from named presets
- use env placeholders only
- flag command-bearing entries as trust-sensitive

Must not:

- write real secrets
- install or execute MCP servers
- emit arbitrary third-party presets outside the supported preset registry

## 13. Testing contract

Each adapter must have:

- unit tests for `supports()`
- snapshot tests for rendered files
- fixture coverage for at least one supported dashboard manifest
- warning coverage for at least one unsupported mapping

Shared tests should also verify that adapters do not emit paths outside the output map.

## 14. Review checklist

Before an adapter is considered complete, it should satisfy this checklist:

- target id is stable
- support checks are explicit
- render output is deterministic
- warnings are actionable
- trust-sensitive files are marked
- no writes occur outside planned paths
- snapshot coverage exists

## 15. Non-goals

This contract does not require:

- runtime plugin loading
- adapter-defined CLI flags
- per-target secret managers
- target-specific repo inspection logic
